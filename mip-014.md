# [MIP-014] Agent Registry + Core Agent Identity Plugin (Minimal)

[Submission details](https://github.com/metaplex-foundation/mip/discussions/52)

# Summary

This MIP outlines an ERC-8004 and x402 compatible Agent Registry for Solana and the Solana Virtual Machine (SVM):

1. An on-chain Agent Registry program that records a pointer to a Metaplex Core asset (by pubkey) via a program PDA, easily indexable by listening to the program or gPAing accounts.

2. A new Metaplex Core ExternalPluginAdapter ("Agent Identity" plugin) that stores a URI to an off-chain agent registration document (JSON) directly on the Core asset, enabling ecosystem composition and indexing.

This design intentionally avoids additional guardrails or opinionated semantics to maximize iteration speed.

# Motivation

"Agents" need a stable identity anchor and a discoverable registration payload (capabilities, endpoints, metadata, etc.), both of which are perfect uses of a high performance and credibly neutral blockchain like Solana.

A minimal, standardized pointer model enables:

- deterministic linking between an "agent" and a Core asset identity,
- consistent off-chain registration documents (ERC-8004-style pattern),
- straightforward indexing/search via existing Core + DAS patterns,
- low-friction iteration while the agent ecosystem is still evolving.

# Goals

- Define a canonical minimal onchain record for agent registration.
- Anchor each agent identity to a Core asset pubkey.
- Point to an offchain registration JSON via a URI.
- Provide a Core plugin surface so apps can resolve agent identity using Core alone, and also provide room for future development of agent behaviors on Core asset lifecycle events.

# Specification

## Terminology

- **Core Asset:** an onchain asset (i.e. NFT) created with mpl-core representing an agent identity

- **Offchain Registration:** a JSON document referenced by URI in mpl-core that describes the agent

## Onchain: Agent Registry Program

**Account:** `AgentIdentityV1`

One record per Core asset.

```
pub struct AgentIdentityV1 {
    /// Account discriminator.
    #[idl_type(Key)]
    pub key: u8,
    /// PDA bump seed.
    pub bump: u8,
    /// Padding for 8-byte alignment.
    #[padding]
    pub _padding: [u8; 6],
    /// The address of the asset.
    pub asset: Pubkey,
}
```

### Derivation

- `agent_record_pda` = PDA(`["agent_identity", core_asset]`)

This makes resolution deterministic by Core asset pubkey.

### Instructions (Minimal)

1. `RegisterIdentityV1(uri)`

- Creates PDA for `core_asset`
- Creates `core_asset` and stores `uri` in the asset data

### Authority / Authorization (Minimal)

To avoid overbearing guardrails while still preventing arbitrary writes:

- `initialize_agent_record` and `update_agent_uri` MUST require a signer that is the core_asset authority or delegate authority.
- The authorization mechanism SHOULD defer to Metaplex Core ownership/authority semantics (e.g., asset owner or asset authority), and be implemented using the canonical Core program checks available at the time of implementation.

No additional permission layers are introduced by the registry.

### Constraints

- `uri` MUST be bounded to prevent account bloat (e.g., max 200 bytes or similar).
- `Key` enables future upgrades without breaking decoders.

## Metaplex Core: Agent Identity ExternalPluginAdapter

This MIP also proposes a new Core ExternalPluginAdapter named AgentIdentity.

```
pub struct AgentIdentity {
    /// URI pointing to the ERC-8004 agent registration JSON file.
    pub uri: String,
}
```

### Intent

- Apps can resolve agent identity using Core alone:
  - read plugin data → get `uri` (and optionally `agent_record` for canonical on-chain reference)
- Indexers can enrich Core asset search results with the agent identity URI.

## Off-chain Registration Document (Non-normative)

The URI points to a JSON payload. This MIP does not lock a final schema, but recommends:

- A top-level `schema` or `version`
- A stable `core_asset` field for self-reference

Flexible fields for endpoints, skills, metadata

Example shape:

```json
{
  "schema": "metaplex.agent.registration.v0",
  "core_asset": "<base58 pubkey>",
  "name": "OpenClaw Agent #123",
  "description": "…",
  "services": [
    {
      "name": "web",
      "endpoint": "https://metaplex.com/agent123"
    },
    {
      "name": "telegram",
      "handle": "Agent123"
    }
  ],
  "capabilities": ["mint", "swap", "post_to_x"]
}
```
